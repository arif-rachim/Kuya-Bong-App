# v0.6 Demo MVP — Implementation Plan

Plan to extend the demo MVP (Vite + React + TS + Zustand, client-only, `localStorage`) for the
consolidated **Blueprint v0.6** (KBA-BP-20260620-003). Builds on the shipped v0.3–v0.4 code.

> Demo constraint: no backend. Photos are compressed client-side and stored as data URLs; "push"
> stays in-app; audit log is local. v0.6 §9 explicitly says: do **not** invent irreversible
> business behavior for credit transfer until open questions are decided — preserve package
> lineage + a full audit trail instead.

## Already shipped (v0.3–v0.4)
Clinic management, announcements, basic Family linking, financial reports (services count / products
income), and a *fixed* master/sub-admin gating model.

## Gap → phases

### Phase 1 — Product Photos (§3) · unblocked, small
- `Product.images?: string[]` (compressed data URLs).
- Client-side resize/compress (canvas, max ~1000px, JPEG ~0.7) before storing; never keep the raw original.
- Upload (1+) + preview in the admin Products create/edit; show thumbnails with product info.

### Phase 2 — Sub-Admin central permission profile + audit log (§6) · rework
- Replace fixed master-only gating with one central `subAdminPermissions` profile (12 toggles):
  Manage Booking, Appointment Mgmt, Clinics, Therapists, Patients, Products, Service Types,
  Cancellation Reasons, Announcements, Follow-Up, Reports-Services, Reports-Products.
- Master-only config panel to edit the profile. Master always has everything; sub-admins use the profile.
- `useCan(capability)` selector drives nav/route guards (replaces `RequireMaster` for capability checks;
  master-only stays for sub-admin management itself).
- **Audit log:** record important master/sub-admin actions (appoint/remove sub-admin, permission changes,
  clinic/therapist/service changes, cancellations, package assignments, credit transfers). New
  `auditLog` collection + a master-only viewer.

### Phase 3 — Friends + Package-Credit Transfer (§5) · large, partially gated
- New `Friend` link (request → notify → confirm → active); kept separate from Family.
- Rules: cannot book for a Friend; cannot use a Friend's package directly.
- **Credit transfer** between confirmed Friends: bidirectional; transferred credit keeps the original
  package expiry; full audit record (sender, recipient, amount, original package ref, expiry).
- Gated decisions (v0.6 §9 Q1–Q4): partial vs full, sessions vs money, re-transfer, reversal. Until
  decided: model the data + audit only; default to **full remaining balance, in sessions, no re-transfer,
  master-reversible via audit** as a clearly-labelled demo assumption (confirm with Kuya).

### Phase 4 — Household Spending & Active Package Report (§8) · medium
- New report: account owner + linked Family; total spending (product purchases), active package
  (start/expiry, total/used/remaining sessions), member per used session, and related Friend credit
  transfers. Friends are not household members (only their transfer records show).

### Minor
- Add `parent`/`sibling` to Family relationship types; keep Family and Friends strictly separate.

## Suggested sequence
1. Phase 1 (photos) — quick visible win
2. Phase 2 (permission profile + audit) — foundational; gates report/announcement access for sub-admins
3. Phase 3 (friends + transfer) — needs §9 answers; build data + audit first
4. Phase 4 (household report) — depends on Family + transfer data

## Open questions (v0.6 §9) to confirm with Kuya
1. Partial credit transfer or full balance only?
2. Credit measured in sessions, money, or both?
3. Can received credit be re-transferred?
4. Can Kuya reverse a transfer?
5. Do sub-admin permission changes take effect immediately?
6. Are reports visible to sub-admins only when the matching permission is on?
7. Announcements to all users or selected?
8. Report sharing PDF-only, or Excel/CSV later?
