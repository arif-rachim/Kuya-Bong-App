## NEW gaps in v0.7

### Gap 1 — Physiotherapist as a role linked to registered users  (BR-09, BR-10; §7.3, §9.3, §13.3) — LARGE
Today "therapists" are free-text master data (`Therapist {id,name,active}`) with no login.
v0.7 requires:
- Physiotherapists are **registered users appointed by the Master Admin** (not free text).
- A physiotherapist can **log in** and see **only their own assigned appointments** ("My Schedule").
- They can cancel/reschedule **their own** appointments (subject to rules) → new actor for cancellation.
- Cannot manage other physiotherapists' appointments unless also a Sub-Admin.
- Master appoints/removes the Physiotherapist role.

### Gap 2 — User deactivation by Master Admin  (BR-03; §8.1) — MEDIUM
- `User` has no `active` flag today. Add one.
- Master can deactivate a user; **deactivated users can't log in or book**.
- Deactivation **preserves** all historical records (appointments, packages, sales, financials).
- If a deactivated user is a Sub-Admin/Physiotherapist, those privileges go inactive.

### Gap 3 — Minor alignments
- **Cancelled by Physiotherapist** status + `cancelledBy: 'physiotherapist'` (status model adds it).
- Rename admin "Therapists" → **"Physiotherapists"** (and select physios from users).
- Clinic **contact details** field (model lists it; today only name + address).
- Family **relationship types** parent/sibling (model lists them; today spouse/child/adult/dependent).
- Admin nav grouping: **Roles** (appoint Sub-Admins & Physiotherapists) + **Physiotherapists** list.

## Fix plan (phased)

### Phase 1 — User deactivation (small, self-contained, do first)
- `User.active?: boolean` (default true); migration backfills `active: true`.
- `deactivateUser(userId)` / `reactivateUser(userId)` (Master only); block deactivating the Master.
- `login`/`loginAs` reject inactive users with a clear message.
- Admin **Patients/Users** screen: Deactivate/Reactivate action (Master only) with confirm + audit log.
- A deactivated user's Sub-Admin/Physiotherapist privileges are treated as inactive in capability checks.

### Phase 2 — Physiotherapist role (linked to users + login + own schedule)
- Extend `Therapist` with `userId?: string` (link to a registered user) — keep `therapistId` on
  appointments/availability to avoid a wide refactor.
- Master **Physiotherapists** screen: appoint a registered user (creates/links a Therapist with
  `userId`), remove, activate/deactivate.
- Role detection: a user is a physiotherapist if an active `Therapist.userId === user.id`.
- New **PhysiotherapistLayout** + routes: **My Schedule** (appointments where the appointment's
  therapist links to the logged-in user), **Appointment Details** (own only), Profile.
- Allow physiotherapist to cancel/reschedule **own** appointments → record `cancelledBy:
  'physiotherapist'` + `CancelledByPhysiotherapist` status; audit log.
- Seed: link a demo physiotherapist user so login can be demoed; persist migration.
- Booking/Calendar therapist pickers continue to work (now backed by appointed physios).

### Phase 3 — Minor alignments
- Add `CancelledByPhysiotherapist` to the status model + badges.
- Rename UI "Therapists" → "Physiotherapists"; ensure selection is user-backed.
- Clinic: add optional `contact` field to model + Clinic Settings form.
- Family: add `parent`/`sibling` to relationship types.

## Open questions to confirm (v0.7)
- Are physiotherapists allowed to cancel/reschedule their own appointments, or view-only? (§7.3 "subject to final business rules")
- Auto-confirm bookings vs manual approval (recommended: auto-confirm).
- Can one user hold multiple roles (e.g., Physiotherapist + Sub-Admin)? (implied yes)

## Suggested sequence
1. Phase 1 (user deactivation) — quick, independent.
2. Phase 2 (physiotherapist role + login) — the main effort.
3. Phase 3 (minor alignments).
Each phase: tsc + build + dual-engine overflow audit, then PR → main.
