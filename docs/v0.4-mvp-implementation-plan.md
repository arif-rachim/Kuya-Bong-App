# v0.4 Demo MVP — Implementation Plan

Plan to update the **demo MVP** (Vite + React + TS + Zustand, client-only, `localStorage`) for
Blueprint Update **v0.4** (KBA-BP-20260620-001). Scope: clinic management, announcements & push,
master/sub-admin roles, financial ad-hoc reports.

> Demo constraint: the app has **no backend**. "Push notifications", "PDF share via phone", and real
> auth are simulated with in-app equivalents; native pieces are flagged as follow-ups (Capacitor/FCM).

---

## Current state (relevant facts)
- `Clinic { id, name, address, active }`. Only `updateClinicName` exists; `ClinicSettings.tsx` edits names.
- Bookings reference `clinicId`; **only appointments + availability** are clinic-linked (packages/purchases are not).
- `Role = 'patient' | 'admin'`; single admin (`u-admin`). `RequireRole` gates patient vs admin.
- Products carry a price (`unitPriceAtSale` snapshot on purchases). **Services have no price field.**
- Patient Home already shows a (non-functional) notification bell — a natural hook for announcements.

---

## Phase A — Clinic Management (low risk)
Maps to v0.4 §1.
- **Store:** `createClinic`, `updateClinic(name,address)`, `toggleClinicActive`, `deleteClinic` with a
  **guard**: block delete if any `appointments` or `availability` reference the clinic → return error
  suggesting deactivation instead.
- **UI:** expand `ClinicSettings.tsx` from name-only to a full list with Add/Edit modal, Activate/Deactivate,
  and Delete (guarded). Reuse the `Products` screen pattern.
- **Booking flows:** ensure `Calendar` and `ManualBooking` only offer **active** clinics for new bookings
  (BookAppointment already filters active).
- Effort: **S–M**. No open-question blockers.

## Phase B — Master Admin & Sub-Admin roles (foundational)
Maps to v0.4 §3. Do this before B/C gating decisions land.
- **Types:** add `adminLevel?: 'master' | 'sub'` to `User` (seed `u-admin` = `master`). Keep `Role='admin'`
  for both so existing route gating still works.
- **Store:** `appointSubAdmin(userId)` (patient → admin/sub), `removeSubAdmin(userId)` (admin/sub → patient);
  never allow changing the master.
- **Gating:** add a `useIsMaster()` selector; hide master-only entries (Service Types, Therapists,
  Cancellation Reasons, Clinic Management, Sub-Admin Management) from sub-admins and guard their routes.
- **UI:** Admin → **Sub-Admins** screen (master only): list users, appoint/remove.
- Effort: **M** (touches nav + a few route guards).
- Depends on open Q2/Q3 (whether sub-admins can see reports / publish announcements).

## Phase C — Announcements & in-app notifications
Maps to v0.4 §2.
- **Types:** `Announcement { id, title, message, createdAt, expiryDate, published }`.
- **Store:** `announcements` state + `createAnnouncement`, `unpublishAnnouncement`; "visible" = published &
  not expired (computed at read time → auto-hide). Seed 1–2 examples.
- **UI:** Admin → **Announcements** (create / list / unpublish / history). Patient → **Announcements** list
  wired to the Home bell icon, showing only active ones (+ unread badge).
- **Native push:** real device push needs Capacitor + FCM/APNs + backend — **out of demo scope**; the
  in-app bell + badge simulates delivery. Flag as a follow-up.
- Effort: **M**.
- Depends on open Q4 (all vs selected customers) — default to **all** for the demo.

## Phase D — Financial ad-hoc reports + PDF share
Maps to v0.4 §4.
- **Income source:** Products use `unitPriceAtSale` (ready). **Services have no price** → to make the
  Services report meaningful, add an optional `price` to `ServiceType` and snapshot it on completed
  appointments (assumption pending Q5).
- **UI:** Admin → **Reports**: filters (Category Services/Products, From=1st-of-month, To=today, Service/Product
  filter). Validate From ≤ To. Output: Services = completed sessions in range; Products = sold products in
  range; show line items + total income.
- **Share as PDF:** demo uses browser **print-to-PDF** (or `jsPDF`) to produce/download the report; native
  share sheet via **Capacitor Share** is a follow-up.
- Effort: **M**.
- Depends on open Q5 (service income recording) and Q6 (PDF only vs Excel/CSV).

---

## Suggested sequence
1. **A. Clinic Management** (independent, quick win)
2. **B. Sub-Admin roles + gating** (unblocks permission scoping for C & D)
3. **C. Announcements** (in-app)
4. **D. Financial reports** (needs service-price decision)

Each phase: typecheck + build green, commit, PR → main (same flow as v0.3 work).

## Open questions to confirm before/while building
1. MVP now, or Phase 2? (affects how much we build for the demo)
2. Can Sub-Admins **view financial reports**? (Phase B/D gating)
3. Can Sub-Admins **publish announcements**, or only Kuya? (Phase B/C gating)
4. Announcements to **all** customers or **selected**? (demo default: all)
5. How is **service income** recorded — manual entry, a service price list, or from package records?
   (demo proposal: add an optional price to ServiceType)
6. Report sharing **PDF only**, or also Excel/CSV later? (demo: PDF via print/jsPDF)

## Out of demo scope (flag as native/backend follow-ups)
- Real push notification delivery (Capacitor + FCM/APNs + backend).
- Native share sheet & on-device PDF file (Capacitor Share/Filesystem).
- Server-side role enforcement (demo enforces in the client only).
